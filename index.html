SGI STL 内存池分配器

仿 SGI STL 二级空间配置器实现，百万级 push_back / pop_back 压测，与标准库性能对比

项目概述

实现一个仿 SGI STL 的二级空间配置器（内存池），并自定义一个简易 vector 容器，对 push_back 和 pop_back 进行百万级数据压测，与 std::allocator 和 std::vector 做性能对比，验证内存池在高频小对象分配场景下的优势。

测试环境

Windows 11 / Clang++ (LLVM) / C++17 / -O2 优化

操作规模

1,000,000 次 push_back + 1,000,000 次 pop_back

数据类型

int（4 字节），验证高频小对象分配性能

核心亮点

SGI 内存池 + 精简容器，比标准库方案快 1.77x

SGIAllocator.h — 二级配置器（内存池核心） 
class PoolAlloc {
private:
    enum : std::size_t { ALIGN = 8, MAX_BYTES = 128, NFREELISTS = MAX_BYTES / ALIGN };
    union FreeNode { union FreeNode* next; char data[1]; };

    static std::size_t round_up(std::size_t bytes) noexcept {
        return (bytes + ALIGN - 1) & ~(ALIGN - 1);
    }
    static std::size_t freelist_index(std::size_t bytes) noexcept {
        return (bytes + ALIGN - 1) / ALIGN - 1;
    }

    // 16 条 free_list + 内存池边界
    static FreeNode* volatile* free_list_base();
    static char*& start_free();
    static char*& end_free();

    // 从内存池取一大块，切分后补充 free_list
    static void* refill(std::size_t n);
    // 内存池核心：分配 nobjs 个大小为 size 的块
    static char* chunk_alloc(std::size_t size, int& nobjs);

public:
    static void* allocate(std::size_t n) {
        if (n > MAX_BYTES) return MallocAlloc::allocate(n);
        FreeNode* volatile* my_list = free_list_base() + freelist_index(n);
        FreeNode* result = *my_list;
        if (!result) return refill(round_up(n));
        *my_list = result->next;
        return result;
    }
    static void deallocate(void* p, std::size_t n) {
        if (n > MAX_BYTES) { MallocAlloc::deallocate(p, n); return; }
        FreeNode* q = static_cast<FreeNode*>(p);
        FreeNode* volatile* my_list = free_list_base() + freelist_index(n);
        q->next = *my_list;
        *my_list = q;
    }
};
MyVector.h — push_back 关键路径 
void push_back(const T& val) {
    if (_finish == _end_of_storage) expand(size() + 1);
    AllocTraits::construct(_alloc, _finish, val);
    ++_finish;
}

// 扩容：2倍增长策略
void expand(size_type min_cap) {
    size_type old_cap = capacity();
    size_type new_cap = old_cap ? old_cap * 2 : 1;
    if (new_cap < min_cap) new_cap = min_cap;

    pointer new_start = AllocTraits::allocate(_alloc, new_cap);
    pointer new_finish = new_start;
    for (pointer p = _start; p != _finish; ++p, ++new_finish)
        AllocTraits::construct(_alloc, new_finish, std::move(*p));

    destroy_range(_start, _finish);
    deallocate_all();
    _start = new_start; _finish = new_finish;
    _end_of_storage = new_start + new_cap;
}
Benchmark.h — 压测引擎 
template <class VecType>
BenchResult run_benchmark(const std::string& name, std::size_t count) {
    BenchResult r;  r.name = name;  r.ops = count;
    VecType vec;

    auto t1 = Clock::now();
    for (std::size_t i = 0; i < count; ++i)
        vec.push_back(static_cast<int>(i));
    auto t2 = Clock::now();
    r.push_ms = Ms(t2 - t1).count();

    t1 = Clock::now();
    for (std::size_t i = 0; i < count; ++i)
        vec.pop_back();
    t2 = Clock::now();
    r.pop_ms = Ms(t2 - t1).count();
    return r;
}

结果分析

SGI Pool vs std::alloc（同容器）— push_back 快 18%
SGI 内存池的 allocate 只需从 free_list 头部取一个节点，是 O(1) 的指针操作。std::allocator 底层走 malloc，每次需要遍历空闲链表，可能触发 brk/mmap 系统调用。内存池一次性向 OS 申请大块内存，后续分配全在用户态完成。
SGI Pool + MyVector vs std::vector — push_back 快 77%
加速来自两个因素叠加：① allocator 优势（约 18%）；② 容器实现差异（约 51%）— MSVC 的 std::vector 有迭代器调试检查、异常安全路径、边界安全检查等额外开销，MyVector 是极简实现。
MyVector vs std::vector（同 alloc）— push_back 快 51%
MyVector 的 push_back 路径极短：判断扩容 → construct → ++_finish。std::vector 路径更长：安全检查 → 异常处理 → 迭代器失效标记 → 实际操作。精简容器实现本身就能带来显著性能提升。
为什么 pop_back 全部接近 0？
pop_back 对 int 来说几乎是空操作：--_finish + 析构（int 是 trivially destructible，编译器直接优化掉）。100 万次指针减法在现代 CPU 上耗时不到 1ms，低于计时精度。

内存分析
理论上 100 万个 int = 4 MB，但实际增量远小于 3 × 4 MB = 12 MB。原因：三组测试顺序执行，前一组 vector 析构后释放的内存被后一组复用；SGI 内存池的 deallocate 把块挂回 free_list 而非归还 OS，可被后续分配复用。1.05 MB 的增量主要来自 SGI 内存池持有的未归还内存（内存池只增不减的特性）。

总结

对比push_back 加速比主要原因
SGI Pool vs std::alloc（同容器）1.18x内存池 allocate 更快，减少系统调用
SGI Pool + MyVector vs std::vector1.77x内存池优势 + 精简容器无额外检查

核心结论：SGI 内存池在高频小对象分配场景下，通过 free_list + chunk_alloc 机制，将内存分配从系统调用级别降低到用户态指针操作级别，配合精简的容器实现，最终实现了比标准库方案快 77% 的性能提升。
